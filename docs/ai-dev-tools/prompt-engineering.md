# 前端开发 Prompt 工程

Prompt 工程是 AI 辅助开发的**核心技能**。同样的 AI 模型，不同的 Prompt 可以让输出质量产生天壤之别。本文提供一套面向前端开发者的完整 Prompt 模板库和实战技巧。

## Prompt 工程在前端开发中的价值

| 场景     | 无 Prompt 技巧           | 有 Prompt 技巧             | 效果差异         |
| -------- | ------------------------ | -------------------------- | ---------------- |
| 组件生成 | 基础结构，缺少边界处理   | 完整组件 + 类型 + 错误处理 | 2-3x 质量提升    |
| Bug 修复 | 表层修复，可能引入新问题 | 根因分析 + 系统性修复      | 减少 80% 回退    |
| 代码重构 | 简单替换，逻辑可能改变   | 保持行为一致的渐进重构     | 安全性大幅提升   |
| 测试编写 | 仅覆盖正常路径           | 覆盖边界 + 异常 + 并发     | 覆盖率 60% → 95% |

## Prompt 编写基本原则

### 四大核心原则

```
┌──────────────────────────────────────────┐
│  1. 清晰具体 — 明确告诉 AI 你要什么      │
│  2. 提供上下文 — 给出相关代码和约束       │
│  3. 分步骤 — 复杂任务拆解为步骤          │
│  4. 指定输出格式 — 明确代码语言和结构     │
└──────────────────────────────────────────┘
```

:::tip 黄金法则
把 AI 当作一个**能力极强但不了解你项目的新同事**——它什么都会写，但需要你提供足够的上下文和明确的指示。
:::

## 代码生成 Prompt 模板库

### React 组件生成模板

```markdown
## Prompt 模板

请创建一个 [组件名称] React 组件：

**功能需求：**

- [需求1]
- [需求2]
- [需求3]

**技术要求：**

- 使用 TypeScript，导出 Props 接口
- 使用 [样式方案: Tailwind CSS / CSS Modules]
- 使用函数式组件 + Hooks

**Props 接口：**

- [prop1]: [类型] - [说明]
- [prop2]: [类型] - [说明]

**边界处理：**

- 空数据时显示 [占位内容]
- 加载中显示 [loading 状态]
- 出错时显示 [错误提示]

**参考：** @file [相关组件文件路径]
```

**实际使用：**

```markdown
请创建一个 NotificationBell React 组件：

功能需求：

- 显示铃铛图标，右上角显示未读数量红点
- 点击展开通知列表（最近 5 条）
- 支持标记已读和全部已读
- 新通知到达时铃铛有动画效果

技术要求：

- 使用 TypeScript，导出 Props 接口
- 使用 Tailwind CSS
- 动画使用 framer-motion

Props 接口：

- notifications: Notification[] - 通知列表
- onMarkRead: (id: string) => void - 标记已读回调
- onMarkAllRead: () => void - 全部已读回调

边界处理：

- 无通知时显示"暂无通知"
- 列表超过 5 条显示"查看更多"链接
```

````
```markdown
## Prompt 模板

请为以下代码编写单元测试：

@file [被测文件路径]

**测试要求：**
- 使用 [Vitest/Jest] + React Testing Library
- 覆盖以下场景：
  - 正常渲染
  - 用户交互（点击、输入、提交）
  - 边界情况（空数据、超长文本、特殊字符）
  - 错误状态
  - 异步操作

**不需要测试：**
- 样式和布局细节
- 第三方库内部逻辑
````

### 代码审查模板

```markdown
## Prompt 模板

请对以下代码进行 Code Review：

@file [代码文件]

审查维度：

- 🔒 安全性（XSS、注入、敏感信息暴露）

- 🏗️ 架构（职责划分、耦合度、可扩展性）

- 严重问题（必须修复）
- 建议改进（推荐修复）
- 优化点（可选优化）

每个问题标注行号和具体修改建议。
```

## 重构建议 Prompt

### 代码简化模板

```markdown
请简化以下代码，保持功能完全不变：

@file [文件路径]

要求：

- 减少代码行数，提高可读性
- 使用现代 JavaScript/TypeScript 特性
- 利用语言内置方法替代手动实现
- 提取重复逻辑为工具函数
- 保持原有的类型安全性

请展示修改前后的对比。
```

### 架构优化模板

```markdown
当前项目结构存在以下问题：[问题描述]

@folder [相关目录]

请提供重构方案：

1. 问题分析（为什么当前结构有问题）
2. 目标架构（推荐的目录和模块划分）
3. 迁移步骤（如何安全地从当前架构迁移）
4. 风险评估（可能的破坏性变更）

约束条件：

- 渐进式重构，不能一次性全改
- 保持向后兼容
- 每步可独立测试和部署
```

### TypeScript 类型改进模板

```markdown
请改进以下代码的 TypeScript 类型：

@file [文件路径]

改进目标：

- 消除所有 any 类型
- 使用泛型提高复用性
- 使用 discriminated union 替代可选属性
- 添加适当的类型守卫
- 确保类型推断友好（减少显式注解需求）

约束：

- 不改变运行时行为
- 不增加不必要的复杂度
- 类型错误信息要对开发者友好
```

## 高级技巧

### Chain of Thought（逐步思考）

```markdown
请逐步分析并实现以下需求：

需求：实现一个支持撤销/重做的文本编辑器状态管理

请按以下步骤思考：

1. 首先，分析需要什么数据结构来支持撤销/重做
2. 然后，设计 API 接口（哪些方法是必需的）
3. 接着，考虑边界情况（历史记录上限、分支操作）
4. 最后，给出完整实现代码

每一步都先解释思路，再给出代码。
```

### 约束条件

```markdown

硬性约束：

- 不使用任何第三方库
- 不能使用 IntersectionObserver polyfill
- 包体积不超过 2KB（gzipped）
```

## 常见反模式

:::warning 避免这些 Prompt 反模式

| 反模式       | 示例                  | 问题                       |
| ------------ | --------------------- | -------------------------- |
| 太模糊       | "帮我写个组件"        | AI 不知道写什么组件        |
| 太冗长       | 500字描述一个简单按钮 | 信息过载，核心需求被淹没   |
| 缺少上下文   | "修复这个 bug"        | AI 不知道 bug 的表现和环境 |
| 期望过高     | "重构整个项目"        | 范围太大，质量无法保证     |
| 结果不可验证 | "写出优雅的代码"      | "优雅"没有客观标准         |
| 忽略现有模式 | 不引用项目代码        | 生成的代码与项目风格不一致 |

:::

## 实战对比：好的 Prompt vs 差的 Prompt

### 案例1：创建表单组件

**❌ 差的 Prompt：**

```
帮我做个登录表单
```

**✅ 好的 Prompt：**

```markdown
创建一个登录表单组件 LoginForm.tsx：

功能：

- 邮箱 + 密码字段
- 客户端验证（邮箱格式、密码最少6位）
- 提交时显示 loading 状态
- 错误信息显示在对应字段下方

技术：

- React Hook Form + Zod 验证
- Tailwind CSS 样式
- 提交回调通过 props.onSubmit 传递

参考项目中已有的表单风格：@file src/components/RegisterForm.tsx
```

### 案例2：性能优化

**❌ 差的 Prompt：**

```
这个页面很慢，优化一下
```

**✅ 好的 Prompt：**

```markdown
@file src/pages/Dashboard.tsx

这个仪表盘页面在数据量大时（>1000条记录）滚动卡顿。

现象：

- Chrome DevTools 显示每帧渲染时间 > 50ms
- React Profiler 显示 DashboardChart 组件频繁重渲染

请分析性能瓶颈并优化：

1. 识别不必要的重渲染
2. 大列表是否需要虚拟化
3. 计算密集操作是否需要 Web Worker
4. 给出具体的代码修改（保持现有 API 不变）
```

:::tip 总结
好的 Prompt = **明确目标** + **足够上下文** + **清晰约束** + **可验证标准**。花 2 分钟写好 Prompt，能节省 20 分钟的来回修改。
:::
