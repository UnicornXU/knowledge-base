# 微前端

> **"微前端是将巨石应用拆分为多个可独立开发、独立部署、独立运行的前端应用的架构方案"** —— 当项目规模增长到一定程度，微前端是解决工程协作与技术栈异构的核心手段。

## 什么是微前端？

```
微前端的本质
═══════════════════════════════════════════════════════

微前端 = 多个独立前端应用的"组合"

核心思想：
• 每个子应用独立开发、独立部署、独立运行
• 主应用负责加载、调度、协调子应用
• 子应用可以使用不同技术栈（React / Vue / Angular）
• 子应用之间可以通信，但尽量松耦合

类比理解：
  后端微服务 → 将后端拆分为多个独立服务
  前端微前端 → 将前端拆分为多个独立应用
```

## 为什么需要微前端？

```
你遇到过这些问题吗？
═══════════════════════════════════════════════════════

❌ 一个项目几十人开发，代码冲突不断
❌ 老项目用 jQuery，新功能想用 React/Vue，但改不动
❌ 改一行代码要等整个项目构建 10 分钟
❌ 一个模块出 Bug 导致整个应用崩溃
❌ 不同团队想用不同技术栈，但被"统一框架"绑死

微前端就是解决这些痛点的架构方案
```

## 微前端知识图谱

```
微前端技术体系
═══════════════════════════════════════════════════════

核心问题
├── 应用加载 ── 怎么把子应用加载到主应用中？
│   ├── 路由劫持
│   ├── 动态 script / link 注入
│   └── Webpack Module Federation
│
├── JS 沙箱 ── 怎么隔离子应用的 JavaScript？
│   ├── 快照沙箱（Snapshot Sandbox）
│   ├── 代理沙箱（Proxy Sandbox）
│   └── iframe 沙箱
│
├── CSS 隔离 ── 怎么防止子应用样式冲突？
│   ├── Shadow DOM
│   ├── Scoped CSS / CSS Modules
│   ├── 命名空间前缀
│   └── 动态样式表切换
│
└── 应用通信 ── 子应用之间怎么传递数据？
    ├── Props 传递
    ├── 自定义事件（CustomEvent）
    ├── 全局状态管理
    └── URL 参数

主流方案
├── single-spa     —— 微前端先行者，路由劫持 + 生命周期
├── qiankun        —— 蚂蚁金服，基于 single-spa 的开箱即用方案
├── Module Federation —— Webpack 5 原生方案，模块共享
├── Micro-App      —— 京东，基于 WebComponent 的轻量方案
├── wujie          —— 腾讯，基于 iframe + Web Components
└── 无界 (Garfish)  —— 字节跳动，沙箱 + 模块化
```

## 三种主流方案对比

| 特性 | single-spa | qiankun | Module Federation |
|------|-----------|---------|-------------------|
| 核心思路 | 路由劫持 + 生命周期 | single-spa + 沙箱 + 预加载 | 模块共享 + 动态远程加载 |
| JS 沙箱 | 无（需自行实现） | Proxy / 快照沙箱 | 无（同域共享） |
| CSS 隔离 | 无（需自行实现） | Shadow DOM / Scoped | 无（需自行处理） |
| 技术栈无关 | 支持 | 支持 | 需同为 Webpack/Rspack |
| 接入成本 | 高（需自行处理沙箱等） | 低（开箱即用） | 中（需配置 Webpack） |
| 共享依赖 | 需自行处理 | 需自行处理 | 原生支持 |
| 适用场景 | 需要高度定制的场景 | 企业级中后台项目 | 同技术栈模块联邦 |
| 维护方 | 社区 | 蚂蚁金服 | Webpack 团队 |

## 学习路线

```
推荐学习顺序
═══════════════════════════════════════════════════════

第一阶段：理解核心概念
  微前端定义 → 应用加载 → JS 沙箱 → CSS 隔离

第二阶段：掌握主流方案
  single-spa（理解原理）→ qiankun（掌握使用）→ Module Federation（了解趋势）

第三阶段：深入原理与实战
  通信方案 → 性能优化 → 落地实践 → 方案选型
```

## 面试考察重点

- **微前端的核心问题**：应用加载、JS 沙箱、CSS 隔离、应用通信
- **qiankun 原理**：Proxy 沙箱实现、样式隔离策略、子应用生命周期
- **single-spa 原理**：路由劫持机制、应用注册与调度
- **Module Federation**：共享依赖机制、动态远程加载原理
- **方案选型**：不同场景下如何选择微前端方案
- **常见陷阱**：全局变量污染、样式冲突、性能问题、多实例共存
