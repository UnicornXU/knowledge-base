# TCP 与连接建立

TCP（Transmission Control Protocol）是互联网核心协议之一，理解其工作机制是前端面试的重要基础。

## OSI 七层模型与 TCP/IP 四层模型

```
  OSI 七层模型           TCP/IP 四层模型       对应协议（前端相关）
 ┌──────────────┐      ┌──────────────┐      ┌────────────────────┐
 │   应用层      │      │              │      │ HTTP, WebSocket,   │
 ├──────────────┤      │   应用层      │      │ DNS, HTTPS(TLS)   │
 │   表示层      │      │              │                    │
 ├──────────────┤      ├──────────────┤      │ TCP, UDP          │
 │   会话层      │      │   传输层      │      │                   │
 ├──────────────┤      ├──────────────┤      ├────────────────────┤
 │   传输层      │      │   网络层      │      │ IP, ICMP, ARP     │
 ├──────────────┤      ├──────────────┤      ├────────────────────┤
 │   网络层      │      │ 网络接口层    │      │ Ethernet, Wi-Fi   │
 ├──────────────┤      │              │      │                   │
 │ 数据链路层    │      └──────────────┘      └────────────────────┘
 ├──────────────┤
 │   物理层      │
 └──────────────┘
```

**前端开发者需要关注的层次：**

| 层次 | 前端关联 |
|------|---------|
| 应用层 | HTTP 请求/响应、WebSocket 通信、DNS 解析 |
| 传输层 | TCP 连接建立与断开、端口号 |
| 网络层 | IP 地址、路由、跨域请求的网络路径 |

## TCP 三次握手

TCP 建立连接需要三次握手，确保双方都能发送和接收数据。

```
    客户端 (Client)                    服务器 (Server)
         │                                  │
         │  ──── SYN (seq=x) ──────────>   │  第1次：客户端发起连接请求
         │                                  │
         │  <── SYN-ACK (seq=y, ack=x+1) ─ │  第2次：服务器确认并回应
         │                                  │
         │  ──── ACK (ack=y+1) ────────>   │  第3次：客户端确认连接建立
         │                                  │
         │        === 连接已建立 ===         │
```

**为什么是三次而不是两次？**

- **防止历史连接**：如果只有两次握手，网络中延迟的旧 SYN 包可能导致服务器误开连接
- **同步双方初始序列号**：双方都需要确认对方的 ISN（Initial Sequence Number）
- **两次握手的问题**：服务器无法确认客户端收到了自己的 SYN-ACK

```javascript
// 浏览器发起 HTTP 请求时，底层自动完成三次握手
fetch('https://api.example.com/data')
  // TCP 三次握手 → TLS 握手 → HTTP 请求 → 数据传输
  .then(res => res.json())
  .then(data => console.log(data));
```

## TCP 四次挥手

TCP 断开连接需要四次挥手，确保双方数据都已传输完毕。

```
    客户端 (Client)                    服务器 (Server)
         │                                  │
         │  ──── FIN (seq=u) ──────────>   │  第1次：客户端请求断开
         │                                  │
         │  <── ACK (ack=u+1) ─────────── │  第2次：服务器确认收到
         │                                  │
         │     （服务器可能继续发送数据）       │
         │                                  │
         │  <── FIN (seq=w) ──────────── │  第3次：服务器请求断开
         │                                  │
         │  ──── ACK (ack=w+1) ────────>   │  第4次：客户端确认断开
         │                                  │
         │   TIME_WAIT (2MSL)               │
         │        === 连接已关闭 ===         │
```

**为什么是四次而不是三次？**

TCP 是全双工通信，每个方向的关闭需要独立确认。当客户端发送 FIN 时，表示客户端不再发送数据，但服务器可能还有数据要发送，所以 ACK 和 FIN 分开发送。

**TIME_WAIT 状态（等待 2MSL）：**

- 确保最后一个 ACK 能到达服务器
- 让旧连接的数据包在网络中消亡，防止影响新连接

## TCP 与 UDP 对比

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接方式 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠传输（确认、重传） | 不可靠（尽最大努力交付） |
| 数据顺序 | 保证顺序 | 不保证顺序 |
| 传输方式 | 字节流 | 数据报 |
| 速度 | 较慢（有开销） | 较快（头部开销小） |
| 流量控制 | 有（滑动窗口） | 无 |
| 首部大小 | 20-60 字节 | 8 字节 |
| 应用场景 | HTTP、邮件、文件传输 | DNS 查询、视频直播、游戏 |

```javascript
// 前端中常见的 TCP 应用
const ws = new WebSocket('wss://example.com');     // 基于 TCP
fetch('https://api.example.com');                   // 基于 TCP

// UDP 在前端较少直接使用，但 WebRTC 使用 UDP
const pc = new RTCPeerConnection();                 // 基于 UDP
```

## TCP 流量控制 —— 滑动窗口

TCP 使用滑动窗口机制进行流量控制，防止发送方淹没接收方。

```
  发送窗口（发送方可发送的数据范围）
 ┌────────┬────────┬────────┬────────┬────────┬────────┐
 │ 已发送  │ 已发送  │ 可发送  │ 可发送  │ 不可发  │ 不可发  │
 │ 已确认  │ 未确认  │ 等待中  │ 等待中  │ 送窗口外│ 送窗口外│
 └────────┴────────┴────────┴────────┴────────┴────────┘
                    ◄─── 窗口大小 ───►
```

- 接收方通过 ACK 报文中的窗口字段告知发送方自己的接收能力
- 发送方根据窗口大小调整发送速率
- 窗口大小动态变化：网络拥塞时缩小，畅通时扩大

## TCP 拥塞控制

TCP 拥塞控制防止过多数据注入网络导致路由器过载。

### 慢启动（Slow Start）

- 初始拥塞窗口 cwnd = 1 MSS
- 每收到一个 ACK，cwnd 翻倍（指数增长）
- 达到慢启动阈值 ssthresh 后进入拥塞避免阶段

### 拥塞避免（Congestion Avoidance）

- cwnd 超过 ssthresh 后改为线性增长（每个 RTT 增加 1 MSS）
- 检测到丢包时，ssthresh = cwnd / 2，cwnd 重新从 1 开始

### 快重传（Fast Retransmit）

- 收到 3 个重复 ACK 立即重传丢失的报文段
- 不必等待超时定时器到期

### 快恢复（Fast Recovery）

- 配合快重传使用：ssthresh = cwnd / 2，cwnd = ssthresh + 3
- 直接进入拥塞避免阶段，而非从慢启动开始

```
 cwnd
  ^
  │         拥塞避免（线性增长）
  │        ╱
  │       ╱  ssthresh（达到阈值）
  │      ╱───────────────────────╲ 发生丢包
  │     ╱                         ╲
  │    ╱  慢启动（指数增长）          ╲ 快恢复
  │   ╱                              ╲──── 重新线性增长
  │  ╱
  │ ╱
  └──────────────────────────────────────────> 时间
```

## QUIC 协议简介

QUIC（Quick UDP Internet Connections）是 Google 开发的传输层协议，是 HTTP/3 的基础。

**核心改进：**

| 特性 | TCP + TLS 1.2 | QUIC |
|------|--------------|------|
| 连接建立 | TCP 3 次握手 + TLS 1-2 次握手 = 3-5 RTT | 1 RTT（首次），0 RTT（后续） |
| 队头阻塞 | 有（TCP 层面） | 无（流级别独立） |
| 连接迁移 | 基于四元组（IP + 端口） | 基于 Connection ID |
| 拥塞控制 | 内核实现 | 用户态实现，可快速迭代 |

```javascript
// 前端无感知，浏览器和服务器自动协商使用 QUIC
fetch('https://example.com'); // 如果服务器支持 HTTP/3，浏览器自动升级
```

## TCP Keep-Alive

TCP Keep-Alive 用于检测空闲连接是否仍然有效。

```
客户端                         服务器
  │                             │
  │  ── Keep-Alive Probe ──>   │  空闲一段时间后发送探测包
  │                             │
  │  <── ACK ──────────────   │  连接仍然存活
  │                             │
  │  ── Keep-Alive Probe ──>   │
  │                             │
  │     (无响应，超时)            │  连接已断开，关闭套接字
```

```javascript
// Node.js 中配置 TCP Keep-Alive
const net = require('net');
const server = net.createServer((socket) => {
  socket.setKeepAlive(true, 60000); // 启用 Keep-Alive，60 秒探测间隔
});
```

## 面试要点

1. **三次握手**：理解每一步的作用，能解释为什么不能用两次
2. **四次挥手**：理解 TIME_WAIT 状态及其存在的意义
3. **TCP vs UDP**：根据应用场景选择合适的协议
4. **拥塞控制**：理解慢启动、拥塞避免的基本原理
5. **QUIC 协议**：了解其相对于 TCP 的改进，特别是 0-RTT 和多路复用

## 常见面试题

**Q: TCP 三次握手的过程是什么？为什么不能两次？**

A: 三次握手：客户端发 SYN → 服务器回 SYN-ACK → 客户端发 ACK。两次握手的问题是服务器无法确认客户端收到了 SYN-ACK，可能建立无效连接；同时无法防止已失效的 SYN 包导致错误连接。

**Q: TCP 和 UDP 的区别？分别适用什么场景？**

A: TCP 面向连接、可靠、有序，适用于 HTTP、文件传输等需要可靠性的场景。UDP 无连接、不可靠、速度快，适用于 DNS 查询、视频直播、在线游戏等对实时性要求高的场景。

**Q: 什么是 TCP 的粘包和拆包？如何解决？**

A: TCP 是字节流协议，没有消息边界。解决方式：(1) 固定长度消息；(2) 消息头包含长度字段；(3) 使用特殊分隔符。HTTP 协议使用 Content-Length 头或 Transfer-Encoding: chunked 来解决此问题。

**Q: 什么是 QUIC 协议？它解决了什么问题？**

A: QUIC 是基于 UDP 的传输协议，解决了 TCP 的队头阻塞问题，将连接建立延迟从多次 RTT 降到 1 RTT（首次）或 0 RTT（后续），支持连接迁移，是 HTTP/3 的基础协议。
