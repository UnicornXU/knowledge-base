# Codex 使用指南

Codex 是面向开发者的 AI 编程助手，可以帮助你理解代码、生成代码、修改文件、解释报错、补充测试和整理文档。对新手来说，它最适合当作一个“会写代码的学习搭子”，而不是完全替你做决定的自动程序。

## 一、Codex 能帮你做什么

| 场景 | 你可以怎么问 | 适合新手吗 |
|------|--------------|------------|
| 解释代码 | “帮我解释这个函数每一段在做什么” | ✅ 非常适合 |
| 修复报错 | “这是报错信息，帮我定位原因并给出修复方案” | ✅ 非常适合 |
| 生成示例 | “用 React 写一个搜索框组件，代码要简单” | ✅ 适合 |
| 补充测试 | “帮这个函数补 3 个单元测试，覆盖边界情况” | ✅ 适合 |
| 重构代码 | “把这段代码拆成更容易读的小函数” | 🟡 需要审查 |
| 大型架构 | “帮我设计完整电商系统架构” | 🟡 需要人工判断 |
| 安全敏感逻辑 | “帮我写登录、支付、权限校验” | ⚠️ 必须人工审查 |

## 二、新手入门流程

### 2.1 先让 Codex 了解背景

不要一上来只说“帮我写代码”。更好的做法是告诉它：项目是什么、你想做什么、希望改哪里、有什么限制。

```text
我在做一个 React + TypeScript 项目。
现在想新增一个用户列表页面，数据先用本地 mock，不接后端。
请先告诉我需要改哪些文件，再给出实现步骤。
```

### 2.2 让它先解释，再动手

新手最容易踩坑的是：AI 改了很多文件，但你不知道为什么。建议先让 Codex 说明计划。

```text
先不要改代码，请先分析这个需求应该怎么实现，列出步骤和可能风险。
```

确认思路没问题后，再让它实现：

```text
按上面的方案实现，保持代码简单，不要引入新的依赖。
```

### 2.3 小步提交任务

不要一次让 Codex 做太大的事情。把任务拆小，效果会更稳定。

```text
第一步：只创建页面结构，不写接口逻辑。
第二步：加入 mock 数据。
第三步：补充搜索和空状态。
第四步：运行构建并修复错误。
```

## 三、常用提示词模板

### 3.1 解释代码

```text
请用新手能听懂的方式解释这段代码：
1. 它的整体作用是什么
2. 每个关键变量代表什么
3. 执行流程是什么
4. 有没有容易出错的地方
```

### 3.2 修复报错

```text
这是我运行项目时的报错：

[粘贴报错信息]

请帮我：
1. 判断最可能的原因
2. 指出应该检查哪些文件
3. 给出最小修改方案
4. 说明为什么这样改
```

### 3.3 生成代码

```text
请帮我实现一个 [功能名称]。
要求：
- 使用 TypeScript
- 代码尽量简单，适合新手阅读
- 不要引入新依赖
- 给出使用示例
- 说明关键代码为什么这样写
```

### 3.4 代码审查

```text
请审查这段代码，重点看：
1. 是否有明显 bug
2. 是否有安全风险
3. 是否有重复逻辑
4. 命名是否清晰
5. 有没有更简单的写法

请按“问题 → 原因 → 修改建议”的格式输出。
```

### 3.5 学习路线

```text
我刚学完 HTML、CSS、JavaScript 基础。
请根据我当前水平，给我一条学习 React 的 4 周路线。
每天安排 1-2 小时，包含练习项目。
```

## 四、写好提示词的 5 个技巧

### 4.1 说明你的水平

```text
我是前端新手，只学过 JavaScript 基础，请用简单语言解释。
```

### 4.2 给出明确目标

```text
目标是让登录按钮点击后跳转到首页，不需要真正接入后端。
```

### 4.3 给出限制条件

```text
不要引入新的 npm 包，只能使用项目已有依赖。
```

### 4.4 要求它先给计划

```text
先列出修改计划，等我确认后再写代码。
```

### 4.5 要求它解释改动

```text
实现后请说明改了哪些文件、每个文件为什么要改。
```

## 五、推荐工作流

```text
理解需求 → 让 Codex 分析方案 → 小步实现 → 运行检查 → 人工审查 → 提交代码
```

### 示例：新增一个搜索功能

```text
我想在文章列表页新增搜索功能。
请按下面流程来：
1. 先阅读相关文件，告诉我当前列表数据从哪里来
2. 给出实现搜索功能的方案
3. 只做最小改动，不要重构无关代码
4. 实现后帮我检查 TypeScript 类型和边界情况
```

这个提示词比“帮我加搜索”更好，因为它说明了目标、范围、步骤和质量要求。

## 六、新手常见误区

### 6.1 误区：完全相信 AI 输出

AI 可能生成看起来正确但实际有 bug 的代码。尤其是接口、权限、支付、登录、文件上传等功能，一定要人工检查。

### 6.2 误区：一次性让它改完整项目

任务越大，AI 越容易偏离目标。新手建议每次只处理一个明确的小功能。

### 6.3 误区：只粘贴一句“报错了”

更好的方式是同时提供：

- 你执行了什么命令
- 完整报错信息
- 相关代码片段
- 你期望的结果
- 实际发生的结果

### 6.4 误区：不运行、不验证

Codex 写完代码后，你仍然需要运行项目、跑测试、看页面效果。AI 生成代码不等于功能已经正确。

## 七、需要重点审查的内容

| 类型 | 为什么要审查 |
|------|--------------|
| 登录鉴权 | 容易出现越权和会话安全问题 |
| 表单提交 | 需要校验输入，避免脏数据 |
| 文件上传 | 可能有大小、类型和路径风险 |
| 接口请求 | 需要处理错误、超时和加载状态 |
| 第三方依赖 | 可能带来体积、安全和维护成本 |
| 性能优化 | AI 可能过早优化或引入复杂写法 |

## 八、一套可直接复制的万能提示词

```text
你是我的编程学习助手。
我现在要完成：[写清楚你的目标]

我的背景：我是新手，只掌握 [写你的技术基础]。
项目技术栈：[写项目技术栈]
限制条件：[例如不要新增依赖、保持代码简单]

请按这个流程帮助我：
1. 先解释你对需求的理解
2. 再列出需要修改的文件和步骤
3. 等我确认后再写代码
4. 写完后解释关键改动
5. 最后告诉我如何验证功能是否成功
```

## 九、学习建议

- 把 Codex 当作“助教”，不要当作“代写工具”
- 每次让它解释至少一个你没看懂的点
- 复制代码前先读一遍变量名和执行流程
- 遇到报错时先让它解释原因，再让它修复
- 每完成一个小功能，就总结这次学到了什么

## 关键点

- Codex 最适合处理明确、小范围、可验证的编程任务
- 好提示词要包含背景、目标、限制、步骤和验证方式
- 新手应该优先让 Codex “解释”和“带着做”，而不是直接生成大段代码
- AI 生成代码必须经过运行、测试和人工审查
- 安全、权限、支付、生产配置等敏感场景不能完全依赖 AI
