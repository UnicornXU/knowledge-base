# 🏗️ 设计模式

> **"设计模式是前人踩过的坑，总结出的最佳实践"** —— 掌握设计模式，能让你写出更优雅、更易维护、更易扩展的代码。

## 为什么前端也要学设计模式？

```
你遇到过这些问题吗？
═══════════════════════════════════════════════════════

❌ 代码到处复制粘贴，改一处要改十处
❌ 一个函数 500 行，谁都不敢动
❌ 组件之间疯狂传 props，层层嵌套
❌ 需求一改，代码要推倒重写
❌ 别人看不懂你的代码，你也看不懂别人的

设计模式就是解决这些问题的"套路"
```

```
设计模式的本质
═══════════════════════════════════════════════════════

设计模式 = 针对特定问题的通用解决方案

它不是语法，不是 API，而是：
• 一种思维方式
• 一种代码组织方式
• 一种团队沟通语言
```

## 设计模式分类

```
23 种经典设计模式（GoF）
═══════════════════════════════════════════════════════

🏗️ 创建型（5种）—— 怎么创建对象
├── 单例模式     —— 全局只要一个实例
├── 工厂模式     —— 批量创建相似对象
├── 建造者模式   —— 分步骤构建复杂对象
├── 原型模式     —— 复制已有对象
└── 抽象工厂     —— 创建一系列相关对象

🔧 结构型（7种）—— 怎么组合对象
├── 适配器模式   —— 让不兼容的接口能一起工作
├── 装饰器模式   —— 动态给对象加功能
├── 代理模式     —— 控制对象的访问
├── 外观模式     —— 提供简化的统一接口
├── 桥接模式     —— 把抽象和实现分离
├── 组合模式     —— 树形结构的统一处理
└── 享元模式     —— 共享细粒度对象节省内存

⚡ 行为型（11种）—— 对象之间怎么通信
├── 观察者模式   —— 一个变了，自动通知其他人
├── 发布订阅     —— 通过中间人解耦
├── 策略模式     —— 算法可以互相替换
├── 命令模式     —— 把操作封装成对象
├── 状态模式     —— 状态决定行为
├── 模板方法     —— 定义算法骨架
├── 迭代器模式   —— 统一遍历方式
├── 中介者模式   —— 用中间人协调多方
├── 备忘录模式   —— 保存和恢复状态
├── 职责链模式   —— 请求沿链传递
└── 访问者模式   —— 在不修改类的前提下添加操作
```

## 学习路线

```
推荐学习顺序
═══════════════════════════════════════════════════════

第一阶段：必学（面试高频 + 日常用）
  单例 → 工厂 → 观察者 → 发布订阅 → 策略 → 代理 → 装饰器

第二阶段：常用（提升代码质量）
  适配器 → 外观 → 命令 → 状态 → 模板方法 → 职责链

第三阶段：进阶（架构设计）
  建造者 → 组合 → 桥接 → 中介者 → 备忘录 → 迭代器
```

## 面试要点

> 设计模式在前端面试中通常以**场景题**的形式出现：「你会用什么模式解决 XXX 问题？」重点掌握：单例、工厂、观察者、策略、代理这 5 个模式的**原理、实现和应用场景**。
