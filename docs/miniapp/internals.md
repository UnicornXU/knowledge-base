# 小程序原理与架构

## 双线程模型

小程序采用**双线程架构**，这是其最核心的设计。逻辑层（JS）和渲染层（WebView）分离，通过 Native 层进行通信。

```
小程序双线程架构
═══════════════════════════════════════════════════════

  ┌───────────────────────────────────────────────────┐
  │                  渲染层（WebView）                  │
  │  ┌──────────┐  ┌──────────┐  ┌──────────────┐    │
  │  │  WXML    │  │  WXSS    │  │  WebView     │    │
  │  │  模板    │  │  样式    │  │  渲染引擎     │    │
  │  └──────────┘  └──────────┘  └──────────────┘    │
  │         │                         │               │
  │         └──── Virtual DOM ────────┘               │
  │                    │                              │
  │              渲染真实 UI                            │
  └────────────────────┬──────────────────────────────┘
                       │
                 Native 层（微信客户端）
                       │
           ┌───────────┼───────────┐
           │  JSBridge 通信        │
           │  setData / 事件回调    │
           └───────────┬───────────┘
                       │
  ┌────────────────────┴──────────────────────────────┐
  │                  逻辑层（JsCore）                   │
  │  ┌──────────┐  ┌──────────┐  ┌──────────────┐    │
  │  │  App     │  │  Page    │  │  JS 逻辑     │    │
  │  │  实例    │  │  实例    │  │  业务代码     │    │
  │  └──────────┘  └──────────┘  └──────────────┘    │
  │         │                                         │
  │    生命周期 / 事件处理 / 数据请求                    │
  └───────────────────────────────────────────────────┘

为什么用双线程？
  ✅ 安全：JS 逻辑无法直接操作 DOM，防止恶意脚本
  ✅ 隔离：业务逻辑崩溃不影响渲染层
  ✅ 管控：平台可以控制 JS 能力（无法访问任意 URL）
  ❌ 代价：通信开销（setData 是性能瓶颈）
```

## 通信机制

```
逻辑层 ↔ 渲染层通信
═══════════════════════════════════════════════════════

  逻辑层 → 渲染层（数据驱动）：
    Page.setData({ count: 1 })
      │
      ▼
    Native 层接收数据
      │
      ▼
    数据 diff（与上次数据对比）
      │
      ▼
    转换为渲染指令（VDOM patch）
      │
      ▼
    WebView 更新 UI

  渲染层 → 逻辑层（事件触发）：
    用户点击按钮
      │
      ▼
    WebView 触发事件（bindtap）
      │
      ▼
    Native 层拦截事件
      │
      ▼
    转发到逻辑层对应的事件处理函数
      │
      ▼
    Page 中的事件回调执行

  通信数据格式：JSON（结构化克隆）
  通信频率限制：频繁 setData 会导致卡顿
```

## 渲染机制

```
小程序渲染流程
═══════════════════════════════════════════════════════

  1. 初始化阶段
     ┌──────────────┐
     │ 加载 WXML    │ → 编译为 JS 模板函数
     │ 加载 WXSS    │ → 编译为 CSS
     │ 加载 JS      │ → 逻辑层初始化
     └──────┬───────┘
            ▼
  2. 首次渲染
     ┌──────────────┐
     │ Page.onLoad  │ → 业务逻辑初始化
     │ Page.onShow  │ → 页面显示
     │ 初始 data    │ → 传入渲染层
     └──────┬───────┘
            ▼
  3. 模板渲染
     ┌──────────────┐
     │ WXML 模板    │ + data → 生成 Virtual DOM
     │ WXSS 样式    │ → 应用样式
     └──────┬───────┘
            ▼
  4. Virtual DOM → 真实 DOM → WebView 显示

  后续更新：
    setData({ key: newValue })
      → 逻辑层计算 diff
      → 传递差异数据到渲染层
      → 渲染层 patch 更新 UI
```

### WXML 编译原理

```
WXML 编译过程
═══════════════════════════════════════════════════════

  WXML 源码：
    <view wx:if="{{show}}">
      <text>{{message}}</text>
    </view>

  编译产物（简化）：
    function _tmpl(data) {
      var h = '';
      if (data.show) {
        h += '<view>';
        h += '<text>' + data.message + '</text>';
        h += '</view>';
      }
      return h;
    }

  编译过程：
    WXML → 解析为 AST → 优化（静态节点标记）→ 生成渲染函数

  关键优化：
    - 静态节点提升（不参与 diff）
    - 列表 key 优化（精确 diff）
    - 模板复用（减少重复编译）
```

## 生命周期

```
应用生命周期（App）
═══════════════════════════════════════════════════════

  App({
    onLaunch() {}   // 应用初始化（全局只触发一次）
    onShow() {}     // 应用显示（从后台切到前台）
    onHide() {}     // 应用隐藏（从前台切到后台）
    onError() {}    // 脚本错误
  })

  触发顺序：
    用户打开小程序 → onLaunch → onShow
    用户切到后台   → onHide
    用户切回前台   → onShow
    用户关闭小程序 → （销毁）

页面生命周期（Page）
═══════════════════════════════════════════════════════

  Page({
    onLoad(options) {}   // 页面加载（接收路由参数）
    onShow() {}          // 页面显示
    onReady() {}         // 页面首次渲染完成
    onHide() {}          // 页面隐藏
    onUnload() {}        // 页面卸载
  })

  触发顺序（页面跳转）：
    A 页面 → B 页面
    A.onHide → B.onLoad → B.onShow → B.onReady

    B 页面返回 A 页面
    B.onUnload → A.onShow

  生命周期流程图：

    打开页面
      │
      ▼
    onLoad ──→ onShow ──→ onReady ──→ [页面活跃]
      │                      │            │
      │                      │            ▼
      │                      │         onHide（跳转其他页面）
      │                      │            │
      │                      │            ▼
      │                      │         onShow（返回）
      │                      │
      ▼                      ▼
    onUnload（页面销毁）
```

### 组件生命周期（Component）

```
Component 生命周期
═══════════════════════════════════════════════════════

  Component({
    lifetimes: {
      created() {}      // 组件实例创建（还不能 setData）
      attached() {}     // 组件进入页面节点树（可 setData）
      ready() {}        // 组件布局完成
      detached() {}     // 组件离开页面节点树
    },

    pageLifetimes: {
      show() {}         // 组件所在页面显示
      hide() {}         // 组件所在页面隐藏
    }
  })

  触发顺序：
    created → attached → ready → [活跃] → detached

  created vs attached：
    created：实例已创建，但未挂载到 DOM，不能 setData
    attached：已挂载到 DOM，可以 setData
```

## 渲染层架构

```
渲染层内部结构
═══════════════════════════════════════════════════════

  ┌──────────────────────────────────────────────────┐
  │                 WebView 容器                      │
  │  ┌────────────────────────────────────────────┐  │
  │  │              Exparser 组件框架               │  │
  │  │  ┌──────────┐  ┌──────────┐  ┌──────────┐ │  │
  │  │  │  <view>  │  │  <text>  │  │ <button> │ │  │
  │  │  │  组件    │  │  组件    │  │  组件     │ │  │
  │  │  └──────────┘  └──────────┘  └──────────┘ │  │
  │  │         │                         │        │  │
  │  │    Virtual DOM 树                  │        │  │
  │  └────────────────────────────────────────────┘  │
  │                     │                            │
  │              真实 DOM 渲染                         │
  └──────────────────────────────────────────────────┘

  Exparser 框架：
    - 小程序自研的组件框架
    - 类似 Web Component（Shadow DOM）
    - 组件间通过 properties / triggerEvent 通信
    - 支持 Component 构造器定义自定义组件

  为什么不用标准 Web Component？
    - 各平台 WebView 对 Web Component 支持不一致
    - 需要额外的能力管控（组件权限、事件拦截）
    - Exparser 可以深度定制，性能更好
```

## 平台差异与适配

```
各平台小程序架构差异
═══════════════════════════════════════════════════════

  微信小程序：
    渲染层：WebView（iOS WKWebView / Android X5）
    逻辑层：JsCore（iOS JavaScriptCore / Android V8）
    通信：Native 中转（evaluateJavascript）

  支付宝小程序：
    渲染层：WebView（自研渲染引擎）
    逻辑层：Nebula JS 引擎
    通信：类似微信

  抖音小程序：
    渲染层：WebView（基于 Chromium）
    逻辑层：V8 引擎
    通信：类似微信

  鸿蒙小程序（快应用）：
    渲染层：原生渲染（非 WebView）
    逻辑层：V8 引擎
    通信：直接调用（无需序列化）

  关键差异：
    ├── JS 引擎不同 → 性能表现有差异
    ├── WebView 版本不同 → CSS/JS 特性支持不一致
    ├── 原生组件实现不同 → 地图、视频等组件表现不一
    └── API 命名空间不同 → wx. / my. / tt.
```

## 性能优化

```
小程序性能优化策略
═══════════════════════════════════════════════════════

1. setData 优化（最重要）
   ├── 减少 setData 频率（合并多次更新）
   ├── 减少 setData 数据量（只传变化的字段）
   ├── 使用路径更新（this.setData({'list[0].name': 'new'})）
   └── 避免在 setData 中传输大对象

   坏的写法：
     this.setData({ list: this.data.list })  // 全量更新

   好的写法：
     this.setData({ 'list[0].name': 'new' })  // 路径更新

2. 渲染优化
   ├── 减少 WXML 节点数量（建议 < 1000 个）
   ├── 减少组件嵌套层级（建议 < 5 层）
   ├── 使用 wx:for 时指定 key
   ├── 长列表使用 recycle-view 组件
   └── 图片使用 lazy-load

3. 代码优化
   ├── 按需引入组件和 API
   ├── 分包加载（主包 < 2MB，总包 < 20MB）
   ├── 预加载分包（preloadRule）
   ├── 公共逻辑提取到自定义组件
   └── 避免在 onShow 中做重计算

4. 启动优化
   ├── 代码包体积优化（移除未使用代码）
   ├── 首屏数据预取（onLoad 中并行请求）
   ├── 骨架屏（减少白屏感知）
   └── 分包预加载

5. 网络优化
   ├── 合并请求（减少请求数量）
   ├── 数据缓存（wx.setStorageSync）
   ├── 图片 CDN + WebP 格式
   └── 接口聚合（减少串行请求）
```
