# 前端架构设计

> **"好的架构让决策可逆"** —— 前端架构不是一成不变的蓝图，而是一套适应变化的决策框架。

## 为什么需要前端架构？

```
小项目（无架构）              大项目（有架构）
┌──────────────┐            ┌──────────────────────────────┐
│  src/        │            │  apps/                       │
│  ├── App.vue │            │  ├── web/          (主应用)   │
│  ├── utils/  │     VS     │  ├── admin/        (管理后台) │
│  └── ...     │            │  └── mobile/       (H5)      │
│              │            │  packages/                   │
│  一个文件搞定 │            │  ├── ui/           (组件库)   │
│              │            │  ├── utils/        (工具库)   │
│              │            │  └── shared/       (共享逻辑) │
└──────────────┘            └──────────────────────────────┘
  3 个月后：💀                  3 个月后：🚀
```

## 核心设计原则

### 1. 关注点分离（Separation of Concerns）

```
┌─────────────────────────────────────────────┐
│                  UI 层                       │
│   组件、页面、样式、动画                      │
├─────────────────────────────────────────────┤
│                业务逻辑层                     │
│   状态管理、业务规则、数据转换                 │
├─────────────────────────────────────────────┤
│                数据访问层                     │
│   API 调用、缓存策略、数据持久化              │
└─────────────────────────────────────────────┘
```

### 2. 依赖方向原则

```
正确方向（依赖倒置）：

  UI 层 ──→ 抽象接口 ←── 数据层
              ↑
           实现层

错误方向（直接依赖）：

  UI 层 ──→ 数据层 ──→ 具体实现
```

### 3. 模块边界清晰

```typescript
// ✅ 好：通过接口通信
interface UserService {
  getUser(id: string): Promise<User>;
  updateUser(id: string, data: Partial<User>): Promise<User>;
}

// ❌ 坏：直接引用实现细节
import { userModel } from '../database/models/user';
```

## 面试考察重点

- **架构选型**：Monorepo vs Multirepo 的取舍、微前端方案对比
- **状态管理**：不同场景下状态管理方案的选择、服务端状态管理
- **路由设计**：嵌套路由实现原理、权限路由的动态生成
- **权限系统**：RBAC 模型设计、按钮级权限实现、动态路由方案
- **设计原则**：SOLID 原则在前端的应用、关注点分离实践
